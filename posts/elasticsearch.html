백엔드 Elasticsearch MySQL

MySQL FullText에서
Elasticsearch로
키워드 검색 개선기

한 글자 검색 불가 + 대량 결과 시 느린 응답 문제를 해결하기 위해 Elasticsearch를 도입하고, nori + ngram 조합으로 검색 속도 4~5배 개선까지의 과정을 기록합니다.

2026.03.26 10 min read Elasticsearch · MySQL · Spring Boot
4~5× 검색 속도 개선
~0.17s 대량 키워드 응답시간
(기존 ~0.8s)

배경 — 왜 바꾸게 됐나

서비스에서 키워드 자동완성 검색을 제공하고 있었는데, 기존 구현은 MySQL FullText 검색을 사용하고 있었습니다. 운영하면서 두 가지 문제가 뚜렷하게 드러났습니다.

문제 1 — 한 글자 검색 불가

MySQL FullText는 기본적으로 최소 토큰 길이(ft_min_word_len) 제한이 있어, "금", "주", "암" 같은 한 글자 키워드를 검색하면 결과가 아예 반환되지 않았습니다. 자동완성 기능 특성상 사용자가 첫 글자만 입력한 시점에도 결과를 보여줘야 하는데, 이게 원천적으로 막혀 있었습니다.

문제 2 — 대량 결과 검색 시 느린 응답

데이터가 늘어나면서 "대통령" 같이 결과 수가 1,000개를 넘는 키워드를 검색할 때 응답이 눈에 띄게 느려졌습니다. 실제로 API를 반복 호출해보니 응답 시간이 700~1,200ms 수준이었습니다.

📌 검색 대상: keywords 인덱스 — 뉴스 관련 키워드 데이터. "대통령" 검색 시 총 1,049개 결과.

1단계 — Elasticsearch 연동 및 성능 비교

Elasticsearch 검색 확인

먼저 ES에 키워드 인덱스를 구성하고, 같은 쿼리를 날려 결과를 확인했습니다.

# "대통령" 키워드로 match 검색, 상위 3개 반환
curl -s 'http://localhost:9200/keywords/_search' \
  -H 'Content-Type: application/json' \
  -d '{
    "query": {"match": {"keyword": "대통령"}},
    "size": 3
  }'

결과는 총 1,049개로 MySQL과 동일하게 나왔고, 상위 결과도 "대통령 대통령실 공사", "대통령 폴란드 대통령과" 등 관련도 높은 키워드들이 잘 잡혔습니다.

실제 서비스 API와 연동 후 성능 측정

Spring Boot 서비스에서 같은 요청을 MySQL과 ES 양쪽 모두 실행하고 시간을 로깅해 비교했습니다.

=== 검색 성능 비교: '대통령' === [MySQL FullText] 1049개 결과, 1193ms [Elasticsearch] 1049개 결과, 310ms 속도 차이: ES가 883ms 빠름 === 검색 성능 비교: '대통령' === [MySQL FullText] 1049개 결과, 776ms [Elasticsearch] 1049개 결과, 138ms 속도 차이: ES가 638ms 빠름 === 검색 성능 비교: '대통령' === [MySQL FullText] 1049개 결과, 754ms [Elasticsearch] 1049개 결과, 131ms 속도 차이: ES가 623ms 빠름

반복 요청할수록 MySQL은 600~1,200ms를 오가는 반면, ES는 꾸준히 130~310ms를 유지했습니다. 전체적으로 약 0.8초 → 0.17초, 4~5배 개선이 확인됐습니다.

엔진평균 응답시간
MySQL FullText~800ms
Elasticsearch~170ms

2단계 — 결과 수가 적을 때는 ES가 느리다?

"주식", "아이폰", "부동산" 등 결과 수가 적은 키워드(10~20개)도 테스트해봤더니 예상 외의 결과가 나왔습니다.

=== 검색 성능 비교: '주식' === [MySQL FullText] 18개 결과, 77ms [Elasticsearch] 18개 결과, 85ms 속도 차이: ES가 8ms 느림 === 검색 성능 비교: '아이폰' === [MySQL FullText] 15개 결과, 20ms [Elasticsearch] 15개 결과, 52ms 속도 차이: ES가 32ms 느림
결과 수가 적을 때는 MySQL이 더 빠를 수 있습니다.

MySQL FullText는 같은 프로세스 내 직접 DB 호출이라 오버헤드가 없는 반면, ES는 HTTP 통신(localhost라도 네트워크 왕복) + JSON 직렬화/역직렬화 고정 비용이 30~50ms 붙습니다. 결과가 소량이면 MySQL이 쿼리를 금방 끝내지만, ES는 이 고정비용이 오히려 발목을 잡습니다.
결과 수유리한 쪽이유
소량 (~수십 개)MySQLHTTP 오버헤드 없음, 직접 DB 호출
대량 (수백~수천 개)Elasticsearch역인덱스 기반 검색이 압도적으로 빠름

3단계 — 갯수 차이가 발생하는 이유 (반도체 케이스)

"삼성" (MySQL 366개 vs ES 355개), "반도체" (MySQL 49개 vs ES 46개)처럼 결과 수가 미묘하게 달랐습니다. 원인을 파고들었습니다.

원인: 한국어 토크나이저의 차이

엔진방식설명
MySQL FullTextBOOLEAN MODE단순 문자열 포함 여부 매칭
Elasticsearchnori 형태소 분석형태소 분해 후 토큰 단위로 매칭

직접 analyze API로 확인해봤습니다.

curl -s 'http://localhost:9200/keywords/_analyze' \
  -H 'Content-Type: application/json' \
  -d '{"analyzer":"korean_analyzer","text":"반도체법"}'

# 결과: tokens → ["반", "도", "체법"]
# "반도체" 토큰이 없음!
핵심 원인: nori가 decompound_mode: mixed로 설정돼 있어 "반도체법"을 반 / 도 / 체법으로 분해합니다. "반도체" 토큰이 생성되지 않아 "반도체법"이 "반도체" 검색 결과에 포함되지 않습니다. MySQL은 단순 문자열 포함이라 "반도체법"에서도 "반도체"를 찾아냅니다.
분석 대상nori 토큰"반도체" 매칭?
반도체반도체, 반, 도체✅ 매칭
반도체법반, 도, 체법❌ 미매칭

4단계 — 해결방법: ngram 서브필드 추가

nori 분석기는 그대로 두되, ngram 서브필드를 추가해 부분 문자열 매칭을 보완하는 방식을 선택했습니다.

쿼리 전략: nori에 높은 가중치, ngram에 낮은 가중치

{
  "query": {
    "bool": {
      "should": [
        { "match": { "keyword":       { "query": "반도체", "boost": 3 } } },
        { "match": { "keyword.ngram": { "query": "반도체", "boost": 1 } } }
      ],
      "minimum_should_match": 1
    }
  }
}
  • nori로 매칭된 결과 → boost 3 → 상위 노출 (정확 매칭 우선)
  • ngram으로만 매칭된 결과 ("한반도", "도체") → boost 1 → 하위 노출
  • min_score 설정 시 점수 낮은 과매칭 결과 자동 필터링 가능

인덱스 크기 실측

인덱스크기비율
nori만 (기존)106.9 MB1x
nori + ngram188.0 MB1.76x
예상했던 2~3배보다 적은 약 76% 증가 (81MB 추가)로, 허용 가능한 수준입니다.

ngram 추가 후 결과

=== 검색 성능 비교: '반도체' === [MySQL FullText] 49개 결과, 141ms [Elasticsearch] 74개 결과, 290ms
과매칭 주의: ngram 추가 후 ES 결과가 74개로 MySQL(49개)보다 오히려 많아졌습니다. "도체 저항 실험", "한반도 평화" 같이 관련 없는 키워드까지 포함된 것입니다. boost + min_score로 관련도 낮은 결과를 잘라내는 튜닝이 필요합니다.

권장 적용 전략

  • 대량 결과 + 한 글자 검색이 필요하다면 → Elasticsearch 사용
  • 결과 수가 적은 정확 매칭 검색은 → MySQL FullText도 충분
  • 복합어 포함 포괄적 검색 → nori + ngram + boost 조합
  • ngram 과매칭 제어 → min_score 또는 상위 N개만 반환
  • decompound_mode: none은 갯수는 맞출 수 있지만 복합어 분해 검색 품질 저하

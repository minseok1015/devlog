MySQL FullText에서 Elasticsearch로 — 키워드 검색 개선기

기존 MySQL FullText 검색의 두 가지 한계를 넘어, Elasticsearch를 도입해 검색 속도를 4~5배 개선하고 한국어 형태소 분석 기반의 정확한 검색 품질을 확보하기까지의 과정을 정리합니다.

배경 — 왜 바꾸게 됐나

서비스에서 키워드 검색을 제공하고 있었는데, 기존 구현은 MySQL FullText 검색을 사용하고 있었습니다. 운영하면서 두 가지 문제가 뚜렷하게 드러났습니다.

문제 1 — 한 글자 검색 불가

MySQL FullText는 기본적으로 최소 토큰 길이(ft_min_word_len) 제한이 있어, "금", "주", "암" 같은 한 글자 키워드를 검색하면 결과가 아예 반환되지 않았습니다.

자동완성 기능 특성상 사용자가 첫 글자만 입력한 시점에도 결과를 보여줘야 하는데, 이게 원천적으로 막혀 있었습니다.

문제 2 — 대량 결과 검색 시 느린 응답

데이터가 늘어나면서 "대통령" 같이 결과 수가 1,000개를 넘는 키워드를 검색할 때 응답이 눈에 띄게 느려졌습니다. 실제로 API를 반복 호출해보니 응답 시간이 700~1,200ms 수준이었습니다.

두 문제를 한꺼번에 해결하기 위해 Elasticsearch 도입을 검토하게 됐습니다.

📌 검색 대상: keywords 인덱스 — 뉴스 관련 키워드 데이터. 대표 키워드 "대통령" 검색 시 총 1,049개 결과.

1단계 — Elasticsearch 연동 및 성능 비교

Elasticsearch 검색 확인

먼저 Elasticsearch에 키워드 인덱스를 구성하고, 같은 쿼리를 날려 결과를 확인했습니다.

# "대통령" 키워드로 match 검색, 상위 3개 반환
curl -s 'http://localhost:9200/keywords/_search' \
  -H 'Content-Type: application/json' \
  -d '{
    "query": {"match": {"keyword": "대통령"}},
    "size": 3
  }'

결과는 총 1,049개로 MySQL과 동일하게 나왔고, 상위 결과도 "대통령 대통령실 공사", "대통령 폴란드 대통령과" 등 관련도 높은 키워드들이 잘 잡혔습니다.

실제 서비스 API와 연동 후 성능 측정

Spring Boot 서비스에서 같은 요청을 MySQL과 ES 양쪽 모두 실행하고 시간을 로깅해 비교했습니다.

[KeywordService] === 검색 성능 비교: '대통령' ===
[MySQL FullText] 1049개 결과, 1193ms
[Elasticsearch]  1049개 결과, 310ms
속도 차이: ES가 883ms 빠름 (MySQL 1193ms vs ES 310ms)

[MySQL FullText] 1049개 결과, 776ms
[Elasticsearch]  1049개 결과, 138ms
속도 차이: ES가 638ms 빠름 (MySQL 776ms vs ES 138ms)

[MySQL FullText] 1049개 결과, 754ms
[Elasticsearch]  1049개 결과, 131ms
속도 차이: ES가 623ms 빠름 (MySQL 754ms vs ES 131ms)

반복 요청할수록 MySQL은 600~1,200ms를 오가는 반면, ES는 꾸준히 130~310ms 대를 유지했습니다.

결과: 약 0.8초 → 0.17초, 4~5배 개선

2단계 — 결과 수가 적을 때는 ES가 느리다?

"주식", "아이폰", "부동산" 등 결과 수가 적은 키워드(10~20개)도 테스트해봤더니 예상 외의 결과가 나왔습니다.

[KeywordService] === 검색 성능 비교: '주식' ===
[MySQL FullText] 18개 결과, 77ms
[Elasticsearch]  18개 결과, 85ms
속도 차이: ES가 8ms 느림

[KeywordService] === 검색 성능 비교: '아이폰' ===
[MySQL FullText] 15개 결과, 20ms
[Elasticsearch]  15개 결과, 52ms
속도 차이: ES가 32ms 느림
⚠️ 결과 수가 적을 때는 MySQL이 더 빠를 수 있습니다.

3단계 — 갯수 차이가 발생하는 이유 (반도체 케이스)

"삼성" (MySQL 366개 vs ES 355개), "반도체" (MySQL 49개 vs ES 46개)처럼 두 엔진의 결과 수가 미묘하게 달랐습니다. 왜 그런지 파고들었습니다.

원인: 한국어 토크나이저의 차이

직접 Elasticsearch의 analyze API로 확인해봤습니다.

curl -s 'http://localhost:9200/keywords/_analyze' \
  -H 'Content-Type: application/json' \
  -d '{"analyzer":"korean_analyzer","text":"반도체법"}'

# 결과: tokens → ["반", "도", "체법"]
# "반도체" 토큰이 없음!
⚠️ 핵심 원인: nori가 decompound_mode: mixed로 설정돼 있어 "반도체법"을 반 / 도 / 체법으로 분해합니다. "반도체" 토큰이 생성되지 않아 "반도체법"이 "반도체" 검색 결과에 포함되지 않습니다. MySQL은 단순 문자열 포함이라 "반도체법"에서도 "반도체"를 찾아냅니다.

MySQL에서는 찾았지만 ES에서 못 찾은 3개:

  1. 당정 반도체법 2월
  2. 반도체법 보조금 재검토
  3. 반도체법 주52시간 예외

4단계 — 해결방법: ngram 서브필드 추가

"반도체법"에서 "반도체"를 찾으려면 부분 문자열 매칭이 필요합니다. nori 분석기는 그대로 두되, ngram 서브필드를 추가해 두 가지를 함께 활용하는 방식을 선택했습니다.

trade-off

장점

  • "반도체법"에서 "반도체" 검색 가능 (부분 매칭)
  • MySQL FullText와 결과 수 일치
  • nori 형태소 분석은 그대로 유지 (검색 품질 보존)
  • 검색 속도 영향 미미 (1~5ms)

단점

  • 인덱스 크기 증가 (ngram 토큰 수가 많아서 약 2~3배 예상)
  • 100만개 기준 초기 동기화 시간 증가
  • ngram 특성상 의도하지 않은 매칭 가능 (예: "도체"만으로도 "반도체" 매칭)
  • 관련성 점수(score) 튜닝이 복잡해질 수 있음

쿼리 전략: nori에 높은 가중치, ngram에 낮은 가중치

{
  "query": {
    "bool": {
      "should": [
        { "match": { "keyword":       { "query": "반도체", "boost": 3 } } },
        { "match": { "keyword.ngram": { "query": "반도체", "boost": 1 } } }
      ],
      "minimum_should_match": 1
    }
  }
}
  • nori로 매칭된 결과 → boost 3 → 상위 노출 (정확 매칭 우선
